\section{Project Scope and Limitations}

\subsection{Major Features}

\subsubsection{Major Features for User}

\begin{itemize}
	\item Register and log in to the system.
	\item Search scientific publications using keywords, authors, journals, or research fields.
	\item View detailed publication information.
	\item Save publications to bookmarks.
	\item Follow research topics of interest.
	\item Receive notifications about new publications and research trends.
	\item View publication trend analysis and statistical dashboards.
	\item Generate and export analytical reports (PDF/CSV).
\end{itemize}

\subsubsection{Major Features for Administrator}

\begin{itemize}
	\item Log in and log out.
	\item Manage user accounts.
	\item Configure academic API sources (Semantic Scholar, OpenAlex, and Crossref).
	\item Synchronize publication data from external APIs.
	\item Monitor the data synchronization process.
	\item Manage system notifications.
	\item Generate system reports.
	\item Configure system settings.
\end{itemize}
\subsection{Project Limitations}

\subsubsection{Limited Use of Full-Text Data}

	\begin{itemize}
		\item The system only uses publication metadata such as title, abstract, keywords, authors, and publication year.
		\item It does not process full-text content, which limits the depth of analysis and understanding of research papers.
	\end{itemize}
	
	\subsubsection{Restricted Data Scope}

	\begin{itemize}
		\item The system focuses only on selected research fields (e.g., Computer Science and Artificial Intelligence).
		\item Therefore, it does not fully represent trends across all scientific disciplines.
	\end{itemize}
		\subsubsection{Dependence on External APIs}

	\begin{itemize}
		\item The system relies on external APIs such as Semantic Scholar, OpenAlex, and Crossref.
		\item Any API downtime, structural changes, or request limitations may affect system stability and data collection performance.
	\end{itemize}
		\subsubsection{Simplified Trend Analysis and Update Frequency}

	\begin{itemize}
		\item Trend analysis is mainly based on publication frequency and growth rate.
		\item It does not apply advanced Machine Learning or AI models for prediction.
		\item Data is synchronized periodically (daily or weekly), so real-time trends may not be immediately reflected.
	\end{itemize}

\section{Product Overview}
	


The Scientific Journal Publication Trend Tracking System (SJPTTS) is a web-based application developed to help researchers, students, lecturers, and academic institutions search, monitor, and analyze scientific publication trends. The system integrates academic data from Semantic Scholar, OpenAlex, and Crossref to provide comprehensive and up-to-date publication information.

The system allows users to search scientific publications using keywords, authors, journals, research fields, or publication years. Users can view detailed publication information, bookmark publications, follow research topics, receive notifications about newly published papers, analyze publication trends through interactive dashboards, and export reports in PDF or CSV format.

Administrators can manage user accounts, configure academic API sources, synchronize publication data, monitor system operations, and maintain the overall performance of the platform. The system aims to provide an efficient solution for discovering scientific literature and supporting research through publication trend analysis.

\subsection{Use Case Descriptions}
\subsubsection{Register}
\textbf{Summary:}
This use case allows a new user to create an account in the Scientific Journal Publication Trend Tracking System. The user provides personal information and account credentials. The system validates the information, creates a new account, and stores the user data for future authentication and access to system functionalities.

\textbf{Flow of Events}

\textbf{Main Flow}
\begin{enumerate}
	\item The user accesses the registration page.
	\item The system displays the registration form.
	\item The user enters the required information, including:
	\begin{itemize}
		\item Full Name
		\item Email Address
		\item Username
		\item Password
	\end{itemize}
	\item The user submits the registration form.
	\item The system validates the entered information.
	\item The system verifies that the username and email address are not already registered.
	\item The system creates a new user account.
	\item The system stores the account information in the database.
	\item The system displays a registration success message.
	\item The system prompts the user to proceed to the Login page.
	\item The use case ends successfully.
\end{enumerate}

\textbf{Alternative Flow}

\textbf{Email Already Exists}
\begin{enumerate}
	\item The user submits the registration form.
	\item The system detects that the email address is already registered.
	\item The system displays: ``This email address is already registered.''
	\item The user enters a different email address.
	\item The main flow resumes from Step 4.
\end{enumerate}

\textbf{Username Already Exists}
\begin{enumerate}
	\item The user submits the registration form.
	\item The system detects that the username already exists.
	\item The system displays: ``This username is already in use.''
	\item The user enters a different username.
	\item The main flow resumes from Step 4.
\end{enumerate}

\textbf{Invalid Input Information}
\begin{enumerate}
	\item The system validates the submitted information.
	\item One or more fields contain invalid data.
	\item The system highlights the invalid fields.
	\item The system displays validation error messages.
	\item The user corrects the information.
	\item The main flow resumes from Step 4.
\end{enumerate}

\textbf{Database Error}
\begin{enumerate}
	\item The system attempts to create the user account.
	\item A database error occurs.
	\item The system displays: ``Unable to create account. Please try again later.''
	\item The account is not created.
	\item The use case ends.
\end{enumerate}

\textbf{Registration Cancelled}
\begin{enumerate}
	\item The user starts the registration process.
	\item The user cancels the registration.
	\item The system returns to the home page.
	\item The use case ends.
\end{enumerate}

\textbf{Special Requirements}
\begin{itemize}
	\item Each email address must be unique.
	\item Each username must be unique.
	\item Passwords shall be stored securely using encryption or hashing mechanisms.
	\item Required fields shall be validated before account creation.
	\item Registration data shall be stored securely.
	\item The registration process should be completed within an acceptable response time.
\end{itemize}

\subsubsection{Synchronize Academic Data}

\textbf{Summary:}
This use case allows the System Administrator to synchronize publication metadata from external academic data sources into the system database. The synchronization process retrieves updated publication information and ensures that the system contains the latest data required for paper searching, trend analysis, dashboard visualization, trending topic identification, and report generation.

The system supports synchronization with external academic APIs such as Semantic Scholar, OpenAlex, and Crossref.

\textbf{Flow of Events}

\textbf{Main Flow}
\begin{enumerate}
	\item The System Administrator logs into the system.
	\item The administrator accesses the Synchronize Academic Data function.
	\item The system displays available academic data sources.
	\item The administrator selects one or more configured API sources.
	\item The administrator initiates the synchronization process.
	\item The system establishes connections to the selected APIs.
	\item The system sends data retrieval requests.
	\item The external APIs return publication metadata.
	\item The system validates the received data.
	\item The system identifies:
	\begin{itemize}
		\item New publications
		\item Updated publication records
		\item Existing records
	\end{itemize}
	\item The system processes the retrieved metadata.
	\item The system inserts new publication records into the database.
	\item The system updates modified publication records.
	\item The system removes duplicate records.
	\item The system records synchronization statistics, including:
	\begin{itemize}
		\item Number of records retrieved
		\item Number of new records added
		\item Number of updated records
		\item Number of duplicate records detected
	\end{itemize}
	\item The system stores synchronization logs.
	\item The system displays a synchronization summary report.
	\item The use case ends successfully.
\end{enumerate}

\textbf{Alternative Flow}

\textbf{API Source Unavailable}
\begin{enumerate}
	\item The system attempts to connect to an academic API.
	\item The API server is unavailable.
	\item The connection request fails.
	\item The system records the error.
	\item The administrator is notified.
	\item The synchronization process continues with other available sources.
	\item The use case continues.
\end{enumerate}

	\item The system cannot retrieve data.
	\item The system displays: ``Synchronization failed due to network connectivity issues.''
	\item The synchronization process stops.
	\item The use case ends.
\end{enumerate}

\begin{enumerate}
	\item The system receives incomplete or malformed metadata.
	\item The system validates incoming records.
	\item Invalid records are rejected.
	\item The system logs validation errors.
	\item Valid records continue to be processed.
	\item The use case resumes from Step 10.

	\item Duplicate entries are ignored or merged.

	\item The system attempts to store synchronized data.
	\item A database error occurs.
	\item The system logs the error.
	\item The administrator is notified.
	\item The synchronization process is terminated.
	\item The use case ends.
\end{enumerate}

	\item No new publication metadata is found.
	\item The system displays: ``No new publication data available.''
	\item The synchronization summary is generated.
	\item The use case ends successfully.
\end{enumerate}

	\begin{itemize}

		\item Crossref API

	\item The synchronization process shall not affect normal user operations.
